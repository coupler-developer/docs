# 2.0.0 릴리스 실행 계획

## 문서 역할

- 역할: `시나리오`
- 문서 종류: `flow`
- 충돌 시 우선 문서: `policy/release-process.md`
- 기준 성격: `transition`

## 목적

- `v2.0.0` 릴리스를 `docs release note 선행 생성 -> RDS -> EC2 -> Mobile -> 최종 태그` 순서로 재현 가능하게 고정한다.

## 범위

- 시작 조건: `docs`, `coupler-api`, `coupler-admin-web`, `coupler-mobile-app`의 릴리스 대상 커밋과 운영 `RDS` 반영 SQL 후보가 확정된 상태
- 종료 조건: `docs`, `coupler-api`, `coupler-admin-web`, `coupler-mobile-app` 태그 발행 완료, 운영 `RDS` Gate 검증 완료, GitHub Release Note 최종화 완료
- 제외 범위: 기능 상세 스펙 검토, 스토어 심사 정책 해석, 신규 마이그레이션 SQL 작성

## 릴리스 상태

- 목표 버전: `v2.0.0`
- 초기 상태: `planned`
- `docs` Release Note 생성 시 상태: `in_progress`
- 모든 반영/검증/태그 완료 시 상태: `released`

## 액터

- Release owner: 전체 순서 통제, GitHub Release Note 최종화
- DB operator: `RDS` baseline/full replay/dev/prod Gate 실행 및 로그 수집
- API/Admin operator: 운영 `EC2` 배포와 smoke test 수행
- Mobile operator: iOS/Android binary 배포, 필요 시 OTA 배포 수행

## 메인 흐름

1. `docs/content/releases/v2.0.0.md`를 `main`에 병합한다.
2. `docs` 레포에서 annotated tag `v2.0.0`을 생성하고 push해 초기 GitHub Release Note를 생성한다.
3. 초기 Release Note 상태를 `in_progress`로 두고, 목표 범위와 체크리스트가 노출됐는지 확인한다.
4. 운영 `RDS` 반영 대상 SQL을 확정한다.
5. `ritzy운영-coupler운영_마이그레이션_가이드/25_EXECUTION_PROCEDURE.md` 기준으로 `운영 dump baseline -> local full replay -> 개발계 검증`을 먼저 완료한다.
6. 운영 `RDS`에는 expand/backfill/cutover-safe 단계만 먼저 반영한다.
7. `DBM-GATE-400` 대상 SQL(`44`, `45`)은 아직 실행하지 않는다.
8. `coupler-api`를 운영 `EC2`에 배포한다.
9. API smoke test를 수행한다.
10. 최소 확인 항목:
    - `/` 응답 확인
    - 핵심 app API 1건 이상
    - 핵심 admin API 1건 이상
    - 운영 로그 10~30분 확인
11. `coupler-admin-web`를 운영 `EC2`에 배포한다.
12. Admin smoke test를 수행한다.
13. 최소 확인 항목:
    - 운영 로그인
    - 회원/심사 핵심 화면 진입
    - 주요 운영 액션 1회
14. `coupler-mobile-app`의 Android/iOS 버전을 `2.0.0`으로 올린 binary를 배포한다.
15. 스토어 심사/배포 반영 후 Mobile smoke test를 수행한다.
16. native 변경이 없는 후속 수정만 OTA로 분리한다.
17. 운영 로그에서 신계약 read/write가 정상인지 확인한다.
18. legacy read/write가 0건인지 모니터링한다.
19. `DBM-GATE-400` 조건을 충족하면 운영 `RDS`에 `44`, `45`를 실행하고 postcheck를 남긴다.
20. `coupler-api`, `coupler-admin-web`, `coupler-mobile-app`에 실제 반영 커밋 기준 annotated tag `v2.0.0`을 생성하고 push한다.
21. `docs` GitHub Release Note 본문을 최종 상태로 수동 갱신한다.
22. 최종 Release Note에 아래를 기록한다.
    - 서비스 최종 태그/SHA
    - 운영 `RDS` 적용 SQL 목록
    - `DBM-GATE-*` 로그 경로
    - 검증 결과
    - 롤백 기준
23. Release 상태를 `released`로 변경한다.

## 예외 흐름

1. `docs` tag push는 성공했지만 서비스 배포가 지연된 경우
   - GitHub Release 상태를 `in_progress`로 유지한다.
   - 서비스 레포 태그는 생성하지 않는다.
   - 릴리스 완료로 간주하지 않는다.
2. 운영 `RDS` precheck 또는 postcheck가 실패한 경우
   - 해당 Gate에서 즉시 중단한다.
   - 운영 반영을 계속 진행하지 않는다.
   - 실패 SQL, Gate ID, 로그 경로를 Release Note 초안에 남긴다.
3. API/Admin 배포는 끝났지만 legacy read/write가 남아 `44`, `45`를 실행할 수 없는 경우
   - contract/drop은 다음 릴리스로 이월한다.
   - `v2.0.0` Release Note에는 `DBM-GATE-400 pending` 상태를 명시한다.
4. Mobile 스토어 배포가 지연된 경우
   - API/Admin 선배포 상태를 유지하되, 구앱 호환 범위 내에서만 운영한다.
   - breaking change가 남아 있으면 롤백 또는 배포 보류를 우선 검토한다.

## 비포함 / 금지

- `docs` 선행 태그를 서비스 레포 태그 완료로 해석하지 않는다.
- `coupler-api`, `coupler-admin-web`, `coupler-mobile-app` 태그를 실배포 전에 생성하지 않는다.
- 운영 `RDS`에 `44`, `45`를 조기 실행하지 않는다.
- `DBM-GATE-*` 근거 없이 릴리스 완료 판정을 하지 않는다.
