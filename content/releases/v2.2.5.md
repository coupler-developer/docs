## 릴리스 결과

- 공통 응답/ErrorData contract cutover 코드는 main 기준으로 준비됐지만 운영 반영은 완료되지 않았다.
- API main은 `response_success`/`response_error` 공통 envelope 경계와 descriptor-first ErrorData 경계를 고정한다.
- Mobile/Admin main은 generated contract copy와 `ok` 기반 응답 boundary를 사용한다.
- 개발계 DB는 `75_expand_admin_manager_nickname.sql`/`76_postcheck_admin_manager_nickname.sql` 적용과 ledger 기록이 확인됐다.
- 운영 DB는 `75/76` 미적용 상태이며, `t_admin.nickname`이 `varchar(20) NOT NULL`이라 이번 expand migration 적용 대상이다.
- 운영 `GET /app/auth/getSettingList?os=google&version_code=1`은 `app_info.force_update: 2`를 반환해 기존 낮은 build 차단 설정은 확인됐다.
- 같은 운영 응답은 아직 top-level `result_code`, `result_msg`, `result_data`를 반환하므로 API 공통 envelope cutover 완료 증거로 사용하지 않는다.

## 메인 흐름

1. API/Admin/Mobile/docs main 최신성과 clean 상태를 확인한다.
2. DB migration SQL을 `coupler-api` repo-relative 경로와 checksum으로 고정한다.
3. 운영 DB read-only preflight로 DB 식별값, ledger, 대상 column contract, 대상 row counter를 확인한다.
4. Admin write 작업을 잠시 freeze한 뒤 운영 DB에서 75/76을 순서대로 적용하고 `schema_migrations` prod row를 기록한다.
5. API/Admin 운영 배포를 진행하고 운영 API 응답이 `{ ok: true, data }` 또는 `{ ok: false, error }` envelope로 전환됐는지 확인한다.
6. Mobile Production NextPush를 Store `2.2.1 (100)` binary target으로 적용한다.
7. Admin 운영자 액션 smoke와 Mobile 앱 기본 진입을 검증한다.
8. rollback 기준과 서비스 태그를 확정한 뒤 docs 릴리스 기록을 `released` 또는 실제 상태로 갱신한다.
