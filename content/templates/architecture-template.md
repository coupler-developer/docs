# Architecture 템플릿

## 문서 역할

- 역할: `설명`
- 문서 종류: `architecture`
- 충돌 시 우선 문서: `<policy 경로 또는 이 문서>`
- 기준 성격: `<as-is | to-be | transition 중 하나>`

전환 중이면 현재 구현과 목표 구조의 차이를 짧게 적는다.

## 목적

- 이 구조 문서가 설명하려는 시스템/도메인을 한 줄로 명시한다.

## 범위

- 포함 범위를 명시한다.
- 제외 범위가 있으면 함께 적는다.

## 상위 규범 문서

- 이 문서가 따르는 `policy` 문서를 링크한다.
- 규범 문서가 없으면 현재 범위에서 왜 `architecture`만으로 충분한지 한 줄로 적는다.
- 구성요소·관계·현재/목표 동작과 불변조건은 설명할 수 있지만 승인·변경 절차, 운영 허용/금지 판정,
  배포 Gate의 원문은 `policy`에 두고 링크한다.
- PR 번호와 소비자 전환 상태 같은 진행 현황은 기술부채 또는 릴리스 기록에 두고 이 문서에는 목표 상태와 추적
  링크만 남긴다. 시점에 따라 바뀌는 API 계약 package의 concrete version은 기술부채에도 복제하지 않고, 특정 릴리스의
  exact version과 기준 ref는 릴리스 기록에 둔다.

## 구성요소

- 핵심 구성요소와 각 책임을 나열한다.

## 논리 데이터 모델

- 데이터를 소유하는 architecture 문서에만 작성한다.
- [논리 데이터 모델 정책](../policy/logical-data-model-policy.md)의 도메인 ID, 표 구조와 정해진 분류값을
  따른다.
- 공통 기능 문서처럼 데이터를 소유하지 않으면 이 절 대신 참조하는 논리 ID와 사용 목적을 설명한다.
- 물리 컬럼·타입·PK/FK·인덱스 목록은 작성하지 않고 중요한 데이터 의미를 책임, 관계, 불변조건으로 표현한다.

- 도메인 ID: `<domain-id>`

### 먼저 보는 그림

- 상세 표를 작성한 뒤 `yarn generate:logical-data-model`을 실행한다.
- 그림과 핵심 규칙은 상세 표에서 자동으로 만들어지므로 직접 고치지 않는다.

## 데이터 흐름

- 입력 -> 처리 -> 저장 -> 출력 흐름을 설명한다.

## 비포함 / 금지

- 이 문서에서 다루지 않는 규칙을 명시한다.
- MUST/SHOULD, 금지 사항의 원문은 규범 문서에 두고 여기서는 링크만 건다.

## 관련 문서

- 연관 `policy`, `fsm`, `flow` 문서를 링크한다.
