# Git 브랜치 전략

## 문서 역할

- 역할: `규범`
- 문서 종류: `policy`
- 충돌 시 우선 문서: 이 문서
- 기준 성격: `as-is`

## 기본 브랜치 구조

- **`main`** → 모든 기능/문서/버그/정리/리팩토링 브랜치가 PR로 합쳐지는 단일 기준 브랜치 (배포 및 태깅)
- **`develop`** → 빈 껍데기처럼 유지하는 임시 실험 브랜치 (필요 시만 사용, main 병합 금지)
- **`feature/*` · `docs/*` · `fix/*` · `chore/*` · `refactor/*`** → 작업 단위별 말단 브랜치

> 별도의 `release/*`, `hotfix/*` 브랜치는 없습니다.
> 모든 배포/패치는 **main에 병합한 뒤 태그(`v1.2.0`, `v1.2.1`)** 로 기록합니다.

## 흐름 요약

```text
 feature/*   docs/*    fix/*    chore/*   refactor/*
     \          |        |         |            /
      \         |        |         |           /
       '--------> main -----------------------> (배포 + 태그)
                   ^
                 tag: v1.2.0 / v1.2.1

 develop (optional sandbox, not merged)
```

## 동작 원리

| 단계        | 작업                              | 브랜치 흐름               | 결과              |
| ----------- | --------------------------------- | ------------------------- | ----------------- |
| ① 작업 생성 | 기능/문서/버그/정리/리팩토링 단위로 브랜치 생성 | `main`에서 `feature/*` 등 | 개발 진행         |
| ② PR 생성   | 코드 리뷰 및 QA                   | `feature/*` 등 `→ main`   | 변경 승인         |
| ③ 병합/태그 | main 병합 후 버전 태그            | `main` + `tag v1.2.0`     | Release 표시      |
| ④ 긴급 수정 | 직접 수정 브랜치 생성 → main 병합 | `fix/* → main`            | `v1.2.x (Hotfix)` |

## 브랜치 이름 규칙

말단 브랜치는 **타입/담당자/주제** 형식으로 작성합니다.

- `feature/박성빈/간편가입`
- `docs/김민식/브랜치전략`
- `fix/박성빈/핫픽스-로그인`
- `chore/김민식/2.0.0_릴리스_이후_정리`
- `refactor/김민식/회원가입_유스케이스_정리`

## Worktree 격리 규칙

- 사용자가 `git worktree`, `새 worktree`, `worktree 파서`처럼 worktree 생성을 명시하면 대상 레포마다 별도 `git worktree`를 생성한다.
- 코드 리뷰, 원인 조사, 영향 분석 자체는 사용자가 지정한 현재 브랜치/PR/커밋에서 읽기 중심으로 진행할 수 있다.
- 재현용 변경이나 수정 후보 검증이 필요하면 기존 작업 브랜치를 직접 수정하지 않고 별도 `git worktree`를 우선 생성한다.
- 별도 worktree의 브랜치도 브랜치 이름 규칙을 따른다. PR 반영 목적이면 `feature/*`, `fix/*`, `docs/*`, `chore/*`, `refactor/*` 중 실제 변경 성격에 맞는 타입을 사용한다.
- 정식 반영 전 임시 조사만 목적이면 `chore/<담당자>/조사-<주제>` 형식을 사용한다. 임시 조사 브랜치는 main 병합 대상이 아니며, 반영이 필요해지면 변경 성격에 맞는 정식 브랜치로 분리한다.
- 조사 결과를 실제 수정으로 반영할 때는 조사 브랜치에서 그대로 진행하지 않고 `fix/*`, `docs/*` 등 변경 성격에 맞는 정식 브랜치/워크트리로 분리한다.
- worktree 경로와 브랜치명은 작업 보고에 남긴다.

## 주의사항

- 브랜치 이름은 **타입/담당자/주제** 형식을 반드시 준수
- PR 병합은 `Rebase and merge`만 사용한다.
- 작업 완료 후 병합된 브랜치는 삭제
- 정기적으로 main 브랜치와 동기화
