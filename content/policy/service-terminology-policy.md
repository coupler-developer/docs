# 서비스 용어 정책

## 문서 역할

- 역할: `규범`
- 문서 종류: `policy`
- 충돌 시 우선 문서: 사용자 노출명과 신규 N:N 그룹미팅 식별자 명명은 이 문서, 기존 배포 계약의 동작과 상태는 각 도메인 정책
- 기준 성격: `transition`
- 전환 범위: Mobile/Admin/docs의 잔존 사용자 노출 문구와 신규 N:N 그룹미팅 계약의 API/Admin/Mobile 소비자 전환
- 완료 조건: 기술부채의 사용자 노출 용어와 그룹미팅 소비자·출시 통합 항목이 각각 완료 기준을 충족해 삭제되고,
  신규 N:N 그룹미팅 계약 식별자가 모든 소비자에서 `group_meeting` 계열로 고정된 상태

## 목적

- UI 노출 용어를 `클럽`, `클럽매니저`, `그룹미팅`으로 정리해 기존 `매칭 매니저`/`큐레이터`/`Ritzy`와
  `로테이션 소개팅` 혼용으로 인한 혼동을 막는다.
- 신규 N:N 그룹미팅 식별자를 API/Admin/Mobile에서 `group_meeting` 계열로 고정해 소비자별 계약 drift를 막는다.

## 적용 범위

- `coupler-mobile-app`, `coupler-admin-web`, `docs`의 사용자 노출 문구, 화면 제목, 버튼, 안내문, 알림 문구
- 신규 N:N 그룹미팅의 DB/API/DTO/endpoint/enum 식별자와 API/Admin/Mobile 소비 경계
- 제외 범위:
    - 이미 구현·배포된 DB 컬럼, API 필드, DTO, endpoint, enum, migration 파일명
    - 운영 도메인, 스토어 SKU, 외부 시스템 키, 레거시 파일/패키지명

## 단일 SoT

- 서비스 용어의 사용자 노출 기준은 이 문서를 단일 기준으로 사용한다.
- 회원 심사 및 `manager_required_auth` 계열 필드 의미는 [회원 심사 단일 정책](member-review-policy.md)을 따른다.
- 잔존 사용자 노출 용어는 [기술 부채 정리](../technical-debt/technical-debt.md)의
  `클럽·클럽매니저 용어 미통일`에서 추적한다.
- 신규 N:N 그룹미팅의 소비자 cutover와 출시 통합은 [기술 부채 정리](../technical-debt/technical-debt.md)의
  `그룹미팅 소비자 cutover 및 출시 통합 미완료`에서 추적한다.
- 기존 `ritzy` 식별자 잔존 정리는 [기술 부채 정리](../technical-debt/technical-debt.md)의
  `ritzy -> coupler 네이밍/운영 잔존 정리` 항목에서 추적한다.

## 용어 사전

| To-be UI 용어 | 의미 | 기존 표현 | 사용 기준 |
| --- | --- | --- | --- |
| `클럽` | 회원이 소속되고 경험하는 프라이빗 매칭/커뮤니티 단위 | `매칭브랜드`, `Ritzy`, `ritzy` | 브랜드, 소속, 공간, 커뮤니티 맥락의 사용자 문구 |
| `클럽매니저` | 클럽별 회원 심사, 매칭 제안, 상담을 담당하는 운영 주체 | `매칭 매니저`, `매칭매니저`, `큐레이터` | 담당자, 상담, 추천, 심사 맥락의 사용자 문구 |
| `그룹미팅` | 한 행사의 다수 참가자가 신청·확정·채팅·후기를 진행하는 모임 기능 | `로테이션 소개팅`, `Rotation Meeting`, `n대n 로테이션 소개팅` | 신규 N:N 사용자 문구와 계약 |

## 필수 규칙

- 회원에게 직접 보이는 문구에서는 `클럽`과 `클럽매니저`를 기본 용어로 사용한다.
- `클럽매니저`는 붙여 쓴다. `클럽 매니저`로 띄어 쓰지 않는다.
- `매칭 매니저`, `매칭매니저`, `큐레이터`는 신규 사용자 노출 문구에 사용하지 않는다.
- `Ritzy`, `ritzy`는 신규 사용자 노출 문구에 사용하지 않는다.
- `manager_id`, `manager_name`, `manager_required_auth`, `manager_required_auth_types`, `curator_status`, `ritzy_comment` 같은 DB/API/DTO 필드명은 UI 문구 전환만으로 변경하지 않는다.
- 신규 그룹미팅의 사용자 노출명은 `그룹미팅`으로 고정하고 `로테이션 소개팅`, `Rotation Meeting`을 신규
  문구에 사용하지 않는다.
- 신규 N:N 그룹미팅의 내부 식별자는 snake_case `group_meeting`, PascalCase
  `GroupMeeting`, 상수 접두사 `GROUP_MEETING_`을 사용한다.
- 기존 2:2 구현의 `t_meeting*`, `/meeting/*`, `MEET_*`는 배포된 레거시 계약이므로 이름만 바꾸지 않는다.
  문서에서는 신규 그룹미팅과 구분하기 위해 `기존 2:2 그룹미팅`으로 표기한다.

## 적용 상태와 Exit Gate

| 상태 | 진입 조건 | 허용 구조 | Exit Gate |
| --- | --- | --- | --- |
| 용어·소비자 전환 중 | 사용자 노출 용어 부채 또는 그룹미팅 소비자·출시 통합 부채가 남아 있음 | 잔존 사용자 표현은 `클럽·클럽매니저 용어 미통일` 범위에서만 유지하고 신규·직접 수정 문구는 표준 용어 사용. 신규 N:N 소비자 차이는 `그룹미팅 소비자 cutover 및 출시 통합 미완료` 범위에서만 유지 | 사용자 노출 잔존 표현 0건과 내부 식별자 예외 분류 완료, API/Admin/Mobile의 신규 N:N 계약 식별자 정렬과 출시 통합 검증 완료, 두 부채 항목 삭제, 이 문서의 기준 성격과 전환 전용 내용을 현재 기준으로 갱신 |
| 최종 상태 | 위 전환 Exit Gate를 통과해 이 문서가 `as-is`로 갱신됨 | 사용자 노출명은 `클럽`·`클럽매니저`·`그룹미팅`, 신규 N:N 식별자는 `group_meeting` 계열만 사용. 기존 2:2 배포 계약은 제외 범위로 유지 | 신규·직접 수정 문구와 계약에서 표준 용어·식별자 검증을 계속 통과하고 미분류 예외 0건 유지 |

- 전환 추적은 `클럽·클럽매니저 용어 미통일`과 `그룹미팅 소비자 cutover 및 출시 통합 미완료`를 각각 사용한다.
  기존 내부 `ritzy` 식별자 분류는 `ritzy -> coupler 네이밍/운영 잔존 정리`로 분리하며 세 항목의 책임을
  서로 완료 근거로 대체하지 않는다.

## 전환 절차

1. 변경 대상에서 사용자 노출 문자열만 먼저 분류한다.
2. `매칭 매니저`, `매칭매니저`, `큐레이터`는 사람/상담/추천 맥락이면 `클럽매니저`로 치환한다.
3. `Ritzy`, `ritzy`, `매칭브랜드`는 클럽/커뮤니티 맥락이면 `클럽`으로 치환하고, 외부 식별자 맥락이면 유지한다.
4. 이미 구현·배포된 코드 필드명과 API 계약명은 변경하지 않는다.
5. 신규 N:N 그룹미팅의 API/Admin/Mobile 소비 식별자를 `group_meeting` 계열로 유지하고, 기존 2:2 운영 식별자는 변경하지 않는다.

## 체크리스트

- [ ] 신규 UI 문구에 `매칭 매니저`, `매칭매니저`, `큐레이터`, `Ritzy`, `ritzy`를 추가하지 않았다.
- [ ] 기존 사용자 노출 문구를 수정할 때 `클럽` 또는 `클럽매니저`로 정리했다.
- [ ] 이미 구현·배포된 DB/API/DTO/endpoint 식별자를 문구 전환 작업에서 함께 바꾸지 않았다.
- [ ] 신규 N:N 그룹미팅 문구와 계약에 `로테이션 소개팅`, `rotation` 식별자를 남기지 않았다.
- [ ] 기존 2:2 `meeting` 운영 계약을 신규 그룹미팅 명칭 변경과 함께 바꾸지 않았다.
- [ ] 사용자 노출 용어 전환과 그룹미팅 소비자·출시 통합을 각각 해당 제목의 기술부채에서 추적했다.
