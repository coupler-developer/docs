# 보안/접근통제 정책

## 문서 역할

- 역할: `규범`
- 충돌 시 우선 문서: 이 문서
- 기준 성격: `as-is`

## 목적

- 인증/인가/권한 변경 흐름을 단일 기준으로 고정해 권한 우회와 운영 실수를 방지한다.

## 적용 범위

- `coupler-api`, `coupler-admin-web`, `coupler-mobile-app`
- 관리자 기능, 운영 액션(승인/반려/제재/무료키 지급), 회원 민감 데이터 조회/수정

## 단일 SoT

- 상위 기술 원칙: [엔지니어링 가드레일](engineering-guardrails.md)
- 심사/권한 도메인 구조: [관리자 권한 시스템](../architecture/admin-permission.md)
- 리뷰/증빙 기준: [코드 리뷰 정책](code-review-policy.md)

## 필수 규칙

### 1) 인증/인가 책임 분리

- 인증(누구인가)은 토큰/세션 검증 미들웨어에서만 수행한다.
- 인가(무엇을 할 수 있는가)는 API 서버에서만 판정한다.
- 클라이언트(Admin/Mobile)는 표시 제어만 수행하고 권한 판정을 우회하지 않는다.

### 2) 서버 단일 판정

- 비즈니스 상태 전이/권한 판정은 API 서버를 단일 기준으로 사용한다.
- Admin UI에서 버튼 숨김 여부와 무관하게 서버에서 동일하게 차단해야 한다.
- `req.admin.super`/담당자 여부 같은 권한 조건은 액션 단위로 명시한다.

### 3) 운영 액션 통제

- 승인/반려/제재/무료키 지급 등 운영 액션은 사유 입력을 필수로 강제한다.
- 중요 액션은 `actor`, `target`, `reason`, `timestamp`, `request_id`를 함께 기록한다.
- 권한 상향(부여)과 권한 하향(회수)은 이력 로그를 남기고, 로그 없는 변경은 금지한다.

### 4) 실패 처리 (Fail-closed)

- 권한 정보 누락/불일치/미정의 상태는 기본 허용이 아니라 즉시 거부한다.
- 예외 분기(`if (...) return;`)로 조용히 통과시키는 코드를 금지한다.
- 권한 오류 응답은 표준 에러 코드/메시지로 노출하고 서버 로그를 남긴다.

### 5) 예외/임시 권한 규칙

- 임시 권한 부여는 담당자, 만료 시점, 추적 이슈를 명시한 경우만 허용한다.
- 만료 조건 없는 임시 권한은 배포 전 차단한다.
- 임시 권한 종료 후 회수 검증 로그를 PR/작업 보고에 첨부한다.

## 운영 절차

1. 권한 변경 요청: 변경 대상/사유/영향 범위를 기록한다.
2. 사전 검토: API 인가 로직, UI 노출 로직, 로그 스키마 변경 여부를 점검한다.
3. 적용: 서버 인가 로직을 먼저 반영하고 클라이언트 표시 로직을 동기화한다.
4. 검증: 권한별 허용/거부 시나리오를 실행해 기대 결과를 확인한다.
5. 롤백: 비정상 권한 노출 시 직전 태그/커밋 기준으로 즉시 복구한다.

## 증빙/추적

- PR 본문에 아래를 필수로 남긴다.
    - 권한 매트릭스 변경 전/후
    - 허용/거부 테스트 결과(로그 또는 캡처 링크)
    - 문서 동기화 반영 경로
- 운영 반영 시 감사 로그 조회 경로를 릴리즈 노트에 명시한다.

## 체크리스트

- [ ] 서버 인가 로직이 클라이언트 표시 로직보다 먼저/동시에 반영됐는가?
- [ ] 권한 없는 사용자 요청이 모두 fail-closed로 차단되는가?
- [ ] 운영 액션(승인/반려/제재/무료키 지급)에 사유 입력/로그가 강제되는가?
- [ ] 임시 권한의 만료 조건/담당자/추적 이슈가 명시됐는가?
- [ ] PR에 권한 테스트 증빙과 문서 동기화 근거를 남겼는가?
